# System Defect Analysis & Remediation Roadmap

> Scope: full source-code audit of the ACPs personalized reading recommendation system as of 2026-02-21, cross-referenced against `PLAN.md` implementation targets.

---

## Part I — Root Cause Synthesis

The individual defects listed below are not independent. They trace back to three structural root causes.

### Root Cause A — The pipeline lacks a real book retrieval step

The current orchestration flows:

```
receive request → analyze user profile → analyze book content → rank books
```

A functional recommender must flow:

```
receive query → retrieve candidate books from a real catalog → analyze user → analyze candidates → rank
```

Because the retrieval step does not exist, every downstream agent operates on either client-supplied book lists or hardcoded stubs (`"seed-001"`, `"seed-002"` in `_derive_books_from_query()`). All agent sophistication — SVD estimation, hash embeddings, scenario policies, explanation LLMs — is applied to fabricated input. The system cannot recommend real books for any query.

### Root Cause B — The LLM serves as a decorator, not as the core matching engine

The LLM is invoked in three places:

| Agent | LLM Task | What it receives |
|---|---|---|
| `reader_profile_agent` | Extract intent keywords | History + reviews (no query) |
| `book_content_agent` | Generate topical tags | Book metadata (no query) |
| `rec_ranking_agent` | Write explanation text | Score numbers (no query, no book description) |

In none of these calls does the LLM receive the user's current query. The LLM — which has substantial world knowledge about books and genres — is never asked to bridge the user's free-text question to relevant book candidates. It is used only for cosmetic post-processing.

### Root Cause C — Profile and content vectors inhabit incompatible spaces

The ranking agent computes `semantic` similarity by calling `_vector_similarity(candidate_vector, user_vector)`. However:

- `candidate_vector` is a 12-dimensional SHA-256 hash embedding produced by `book_content_agent` (see Root Cause D6 below for why this is a hash, not a real embedding).
- `user_vector` is produced by `_flatten_profile_to_vector()`, which packs genre/theme/format distribution weights into a flat float array.

These two vectors are generated by entirely different processes, have no shared semantic space, and their cosine similarity is mathematically meaningless. Even if a real SentenceTransformer model were loaded, the book vectors would be 384-dimensional transformer outputs while the user vector would remain a hand-packed weight array of variable length — the mismatch would persist.

## Part II — Defect Inventory

### D1 — `_derive_books_from_query()` returns hardcoded stubs regardless of query

**Location:** `reading_concierge/reading_concierge.py`, lines 414–442

**Symptom:** When no `books` or `candidate_ids` are provided by the client, the function returns two fixed stubs with IDs `"seed-001"` and `"seed-002"`, titles `"Seed Book One"` and `"Seed Book Two"`, and generic genre labels. The query text is placed into the description string but has zero effect on which books are selected.

**Impact:** The entire recommendation output is predetermined. Changing the query from "recommend a science fiction novel" to "recommend a cookbook" produces the same two seed books.

### D2 — User query is absent from `profile_payload`

**Location:** `reading_concierge/reading_concierge.py`, lines 579–584

**Symptom:** The `profile_payload` dictionary sent to `reader_profile_agent` contains `user_profile`, `history`, `reviews`, and `scenario`, but not `query`. Inside the profile agent, the LLM prompt for intent extraction (`_generate_intent_keywords()`, lines 308–322) is built entirely from history titles and review text. The user's current question is never seen.

**Impact:** Intent keywords reflect historical taste rather than current need. A user with a history of reading fiction who asks "recommend a Python programming book" will receive fiction-oriented intent keywords.

### D3 — Embedding model name `"qwen-plus"` silently triggers hash-vector fallback

**Location:** `agents/book_content_agent/book_content_agent.py`, line 34; `services/model_backends.py`, lines 54–83

**Symptom:** `BOOK_CONTENT_EMBED_MODEL` defaults to `"qwen-plus"` (an LLM chat model, not a SentenceTransformer). The function `_resolve_sentence_transformer("qwen-plus")` throws an exception and returns `None`. Execution silently falls through to the `hash_embedding()` fallback, which produces SHA-256-derived float arrays. These carry no semantic meaning.

**Impact:** The `semantic` dimension of the composite ranking score is computed over pseudo-random vectors. It contributes noise, not signal.

### D4 — `_tokenize_text()` drops all non-ASCII characters (Chinese, Japanese, etc.)

**Location:** `agents/rec_ranking_agent/rec_ranking_agent.py`, lines 131–133

**Symptom:** The regex `r"[a-z0-9]+"` matches only lowercase Latin letters and digits. Any Chinese, Japanese, Korean, Cyrillic, or accented-character query produces an empty token set. `_query_candidate_alignment()` then returns `0.0` for every candidate.

**Impact:** The query-alignment feature (added in the 2026-02-21 fix) is non-functional for any non-English input. No error or warning is raised; the failure is completely silent.

### D5 — User query is absent from `content_payload`

**Location:** `reading_concierge/reading_concierge.py`, lines 586–592

**Symptom:** The `content_payload` dictionary sent to `book_content_agent` does not include `query`. Inside the content agent, `_heuristic_tags_for_book()` computes tag weights from the book's own description and genre fields, completely independent of what the user asked.

**Impact:** Tag relevance cannot be conditioned on user intent. A book about "the history of science fiction" will receive identical tags whether the user asked about "history" or about "science fiction."

### D6 — Explanation LLM prompt lacks query, book description, and user context

**Location:** `agents/rec_ranking_agent/rec_ranking_agent.py`, lines 366–377

**Symptom:** The explanation prompt sent to the LLM contains only `book_id`, `title`, `score_parts`, and `composite_score`. It does not include the user's query, the book's description, the book's genres, or the user's preference profile. The LLM is asked to "explain recommendation rationale" from numerical scores alone.

**Impact:** Generated explanations are generic restatements of score numbers (e.g., "This book scored high on semantic alignment") rather than meaningful natural-language rationales (e.g., "Because you enjoy ecological science fiction, Dune's themes of planetary ecology and political intrigue are a strong match").

### D7 — `_normalize_score_rows()` collapses ranking signal when candidate pool is small

**Location:** `agents/rec_ranking_agent/rec_ranking_agent.py`, lines 206–219

**Symptom:** Min-max normalization is applied per dimension across all candidates. When the pool contains only 2 candidates (as is typical with the hardcoded stubs), one always normalizes to `0.0` and the other to `1.0` on every dimension. With pools of 1 candidate, the raw value is returned unnormalized. This makes the composite score extremely sensitive to which two stubs happen to be in the pool, and insensitive to the actual magnitude of the underlying signals.

**Impact:** Ranking degenerates to an artifact of normalization arithmetic rather than reflecting genuine relevance differences.

### D8 — Session memory grows without bound (memory leak)

**Location:** `reading_concierge/reading_concierge.py`, line 63

**Symptom:** The coordinator stores all sessions in a module-level `dict` (`sessions: Dict[str, Dict[str, Any]] = {}`). There is no eviction policy, no TTL, and no maximum size. Every request that supplies a new `session_id` (or omits it, causing a UUID to be generated) adds an entry that is never removed.

**Impact:** In any long-running deployment or benchmark loop, memory consumption grows monotonically. Under the Phase IV benchmark with 8 cases per iteration, each run leaks 8 session objects containing full `partner_results` payloads.

### D9 — `call_openai_chat()` uses the synchronous OpenAI client inside an `async def`

**Location:** `base.py`, lines 105–133

**Symptom:** The function is declared `async def` but calls `openai.chat.completions.create()` — the synchronous API. It then checks `inspect.isawaitable()` as a fallback, but under normal installation the sync client returns a non-awaitable `ChatCompletion` object. This means the call blocks the asyncio event loop for the entire duration of the HTTP round-trip to the LLM provider.

**Impact:** When the coordinator dispatches profile and content agents in parallel via `asyncio.gather()`, the intended concurrency is defeated. Each LLM call blocks the event loop, serializing what should be parallel work and increasing end-to-end latency.

### D10 — Cold-start seeded history feeds fabricated data into the profile agent

**Location:** `reading_concierge/reading_concierge.py`, lines 456–479, 489–493

**Symptom:** In `cold` scenario, `_seed_cold_start_history()` generates synthetic history entries from the same hardcoded stubs (D1). These entries have a fixed rating of `3`, genre `["fiction"]`, and format `"unknown"`. The profile agent then treats this fabricated data as real user history and derives genre weights, format preferences, and difficulty distributions from it.

**Impact:** Cold-start users always receive a profile biased toward `fiction` with neutral ratings, regardless of their query. The cold-start policy masks the absence of real data rather than exposing it to the recommendation logic.

### D11 — Baseline rankers are heuristic stubs, not real baseline implementations

**Location:** `services/baseline_rankers.py`

**Symptom:** `traditional_hybrid_rank()` and `multi_agent_proxy_rank()` use handcrafted scoring formulas with fixed magic-number weights (e.g., `0.35`, `0.1`, `0.2`). They do not implement actual collaborative filtering, matrix factorization, or any trained model. They operate on the same hardcoded candidate pool as the ACPs pipeline.

**Impact:** The Phase IV benchmark comparison ("ACPs vs. traditional hybrid vs. multi-agent proxy") compares three heuristic systems against each other, all operating on fabricated data. The resulting metrics and leaderboard rankings do not constitute a valid empirical evaluation as required by PLAN.md Section (四).5.

### D12 — `.env` exposes a live API key in plaintext

**Location:** `.env`, line 3

**Symptom:** The file contains `OPENAI_API_KEY=sk-6bfa89621b0a4631b06f4b1adfb6b6ed` — a real DashScope API key committed to the repository (the file is listed as untracked in `.gitignore` status but is present in the workspace).

**Impact:** Security risk. If this file is committed or shared, the API key is exposed. The `.env` file should be excluded via `.gitignore` and replaced with a `.env.example` containing placeholder values.

## Part IV — Prioritized Remediation Roadmap

The following steps are ordered by dependency and impact. Each step unlocks meaningful progress for the steps that follow.

### Priority 1 — Add LLM-backed book retrieval in the coordinator (fixes D1, Root Cause A)

Convert `_derive_books_from_query()` to an `async` function that calls `call_openai_chat` with a structured prompt:

```
Given the user query: "{query}"
Recommend 5–8 real books. Return a JSON array where each element has:
  book_id, title, author, description (2–3 sentences), genres (list of strings).
Only return the JSON array, no other text.
```

This is a temporary substitute for a real catalog lookup, but it makes the output genuinely query-dependent and produces real book metadata for downstream agents to process.

### Priority 2 — Pass `query` into both `profile_payload` and `content_payload` (fixes D2, D5)

Add `"query": req.query` to both payload dictionaries in `_orchestrate_reading_flow()`. Then:

- In the profile agent, include the query in the LLM prompt for `_generate_intent_keywords()` so that intent reflects current need, not just historical taste.
- In the content agent, use query tokens to boost tag relevance scores for books whose metadata overlaps with the query.

### Priority 3 — Fix the embedding pipeline (fixes D3, Root Cause C)

1. Set `BOOK_CONTENT_EMBED_MODEL` to a valid SentenceTransformer model (e.g., `"all-MiniLM-L6-v2"`) in `.env`.
2. In the ranking agent, replace `_flatten_profile_to_vector()` with a sentence-based encoding of the user profile (e.g., `"A reader who prefers science fiction, fast pacing, intermediate difficulty, and themes of ecology and politics"`). Encode this sentence with the same SentenceTransformer model used for book content vectors so that cosine similarity is computed in a shared semantic space.

### Priority 4 — Fix `_tokenize_text()` for Unicode support (fixes D4)

Change `re.findall(r"[a-z0-9]+", text)` to `re.findall(r"[\w]+", text, re.UNICODE)` and remove the `len(tok) >= 3` filter (which would discard valid two-character Chinese words). For production Chinese support, consider integrating `jieba` segmentation.

### Priority 5 — Enrich the explanation prompt (fixes D6)

Add the user's query, the book's description, the book's genres, and a summary of the user's preference profile to the LLM explanation prompt. This allows the model to generate contextual rationales rather than score restatements.

### Priority 6 — Fix `call_openai_chat()` to use the async OpenAI client (fixes D9)

Replace `openai.chat.completions.create()` with `openai.AsyncOpenAI().chat.completions.create()` (or use `await` with the async client). This unblocks the event loop during LLM calls and restores true parallelism in the coordinator's `asyncio.gather()` dispatch.

### Priority 7 — Add session eviction policy (fixes D8)

Add a TTL-based or LRU eviction mechanism to the `sessions` dictionary. At minimum, cap the dictionary size and evict the oldest entry when the limit is reached.

### Priority 8 — Secure the API key (fixes D12)

Remove the real API key from `.env`, add `.env` to `.gitignore`, and provide a `.env.example` with placeholder values.

### Priority 9 — Connect a real dataset (fixes Section (一) gaps)

Obtain the Book-Crossing or Goodreads-10k dataset and implement preprocessing scripts. This is a prerequisite for meaningful evaluation, real collaborative filtering, and knowledge graph construction — all of which are required by PLAN.md Phase 2.

### Priority 10 — Replace baseline stubs with real implementations (fixes D11, Section (四) gaps)

Implement a genuine collaborative filtering baseline (e.g., SVD on a user-item matrix from the real dataset) and, if feasible, integrate or cite MACRec/ARAG results. Without this, the benchmark comparison section of the thesis has no empirical validity.
